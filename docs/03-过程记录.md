# 03-过程记录

**目的**：如实记录本项目在迭代中的**思考方式、关键取舍、学习与调优**，便于课程/答辩中展示**问题拆解能力**与**持续改进**；与 `02-说明文档.md`（对外说明）、`TECH_IMPLEMENTATION.md`（技术复刻）互补——本文侧重**为什么这样做**与**怎么学会的**。

---

## 一、问题意识与阶段目标

最初目标可以概括为：在 **Serverless 友好** 的前提下，搭一套「组织 → 项目 → 任务」协作工具，并具备**文件交付**、**可选 AI 辅助**、**多人协同时的数据刷新感知**。  
实施中很快遇到三类张力：

1. **平台限制 vs 用户体验**：托管函数对请求体大小、运行时间有上限，不能直接照搬「传统后端一次 POST 吞掉整个文件」。  
2. **正确性 vs 可感知性**：接口失败若只写控制台或清掉前端 state，用户会认为「点了没反应」。  
3. **功能广度 vs 可维护性**：实时同步、私信、报表、AI 若全部强依赖外部服务，缺配置时应**降级**而非整站不可用。

下文中的决策，多数是在这三条张力下做的权衡。

## 二、重要决策（技术选型、流程、交互与效果）

### 2.1 技术选型

| 决策 | 选项对比（简述） | 为何如此决定 |
|------|------------------|--------------|
| **框架** Next.js App Router | vs 纯 SPA + 独立后端 | 前后端同仓、Route Handler 即 API，部署路径单一；SSR/布局适合组织级导航。 |
| **数据库** PostgreSQL + Prisma | vs SQLite 长期单机 | 上线与多环境迁移必须可重复；Prisma 迁移与类型生成降低心智负担。 |
| **会话** JWT + HttpOnly Cookie | vs 纯 localStorage Token | XSS 风险更低；与「记住我」天数映射到 Cookie `maxAge` 直观。 |
| **实时** SSE + 进程内 EventEmitter | vs 首屏即 WebSocket | Serverless/网关场景下 SSE 更易通；单机内协作足够；**多副本需 Redis**，在代码里留言避免误以为「天然分布式」。 |
| **对象存储** Supabase Storage（service role） | vs 仅存数据库 BLOB | 大文件不应进 Postgres；签名 URL 便于预览与下载权限控制。 |
| **AI** OpenRouter | vs 直连某一厂商 SDK | 模型可配置、切换成本低；密钥仅服务端，失败时可局部降级。 |

### 2.2 流程与架构

| 决策 | 核心判断 |
|------|----------|
| **交付物上传改为「签发 URL → 浏览器 PUT → 服务端写库」** | Vercel 等对 **单次请求体**约 4.5MB 上限；multipart 经 Next 中转会在稍大文件上**静默失败或网关截断**。直传把「体积流量」挪到存储域名，API 只处理 JSON 元数据。 |
| **GET 交付物列表成功后，单独维护「上传结果提示」，不在 `load()` 开头无脑 `setError(null)`** | 曾出现：上传失败后立刻 `await load()` 刷新列表，`load` 一开就把错误状态清掉，用户**看不到失败原因**。后将「列表错误」与「上传 Toast」拆分。 |
| **权限统一走 `requireProjectAccess` + 组织/项目角色组合** | 支持「仅被拉进项目、未进业务组织」的访客协作，避免模型与现实团队结构脱节。 |

### 2.3 交互与体验

| 决策 | 权衡 |
|------|------|
| **文件选择：禁用 `input[type=file]` 会带来「点了没反应」** | 在「存储未配置」时若 disable 输入框，部分浏览器下 **label / programmatic click 均无法弹出系统文件框**。改为仍可点选，提交时再提示配置；并用 **`label htmlFor`** 提升 Safari/移动端兼容性。 |
| **上传结果：顶部横幅 success/error + 自动消失 + 可手动关闭** | 仅靠列表下方一行红字，在侧栏滚动区域内**易被忽略**；横幅 + `role="alert"`/`aria-live` 兼顾可视与无障碍。 |
| **登录页：浅灰底 + 标题分色 + 蓝色 Tab 选中态** | 与品牌红主按钮区分层次；选中态用蓝色避免与主 CTA 抢「红色唯一强调」。 |
| **工作台「资源中心」入口放在项目顶栏 AI 旁；工作台项目管理页去掉跨项目资源中心大图** | 减少重复心智：「文件归属项目」→ 入口跟项目走；全局汇总块占用纵向空间且与项目内页重复，按反馈收敛。 |

### 2.4 效果与范围

| 决策 | 说明 |
|------|------|
| **未配置 Supabase 时 API 返回 503 + 明确文案** | 优于静默失败或假成功；前端据此禁用或提示，避免用户误以为「网络坏了」。 |
| **诊断接口 `/api/diagnostics`** | 把 DB、JWT、OpenRouter、**存储桶是否存在** 汇总成 JSON，缩短线上排障路径。 |
| **技术说明书独立成文 `TECH_IMPLEMENTATION.md`** | 使用者文档与复刻者文档分离，避免单文件过长、读者错位。 |

## 三、根据使用情况做的调优（实录节选）

下列条目来自真实使用与协作反馈，对应仓库中的提交线索（顺序不分先后）：

1. **交付物列表加载失败时，勿把「存储未配置」写死为 false**  
   否则界面长期禁止上传；应区分「接口失败」与「明确未开通」。

2. **上传链路改为 Supabase 直传 + `complete-upload`**  
   解决大文件与平台体积极限；`complete` 侧对存储列表做短重试，缓解列表最终一致延迟。

3. **上传成功/失败/部分成功的 Toast**  
   解决「打开文件对话框选了文件却像没反应」——其中一部分是状态被 `load()` 清掉，一部分是需要**显式成功反馈**。

4. **登录页视觉回退到指定稿**（灰底、分色标题、蓝 Tab、蓝勾选、分隔线「或」）。

5. **项目管理页移除大块「跨项目资源中心」**  
   保留项目内 `/project/.../assets` 资源库；避免入口重复与页面过长。

6. **文档**  
   增补 `TECH_IMPLEMENTATION.md`，便于学习者复刻；与用户指南、说明文档分工。

若答辩需「是否可持续改进」：**多实例 SSE**、**直传回调签名校验加强**、**上传 virus scan** 等可作为后续课题——当前实现优先 **可落地与可解释**。

## 四、学习过程（已知基础 · 新知识 · 学法）

### 4.1 已有基础（本次主要巩固）

- HTML/CSS 布局、React 函数组件与 hooks、`fetch` 与表单处理。  
- Git 基本分支与推送、环境变量意识。  
- REST 风格接口与「前后端字段对齐」的习惯。

### 4.2 新知识或加深理解的部分

| 主题 | 学到什么 | 怎么学的 |
|------|-----------|----------|
| **Prisma + Neon/Vercel** | `DATABASE_URL`（pooler）与 `DIRECT_URL`（migrate）分工不当会导致迁移超时或锁问题 | 读 Prisma 文档「Deploy to Serverless」、对照报错日志与 `DEPLOY.md` |
| **Next App Router** | Server Component vs `"use client"`、Route Handler 与缓存、`cache: 'no-store'` 对认证接口的意义 | Next 官方文档 + 本地 Network 面板验证 |
| **SSE** | `ReadableStream`、`text/event-stream`、断线重连；与业务 `EventEmitter` 拼接时的**单机边界** | 阅读本项目 `stream/route.ts` + MDN EventSource |
| **Supabase Storage** | `createSignedUploadUrl` / PUT 直传、`createSignedUrl` 预览；桶未创建时的典型 HTTP 报错 | Supabase Storage API 文档 + 诊断接口实测 |
| **Sharp + 头像** | 服务端压图、JPEG 质量阶梯、data URL 上限与 Postgres TEXT | 试错 + 控制 `MAX_DATA_URL_LEN` |
| **无障碍与表单** | `aria-live`、`role="alert"`、禁用 file input 对「假死」交互的影响 | WebAIM 摘要 + 真机点测 |

### 4.3 主动探索（非教程逐步带着做的）

- **为何上传「无反馈」**：不止查 Network，而是**追踪 React state 生命周期**，发现 `load()` 清空错误状态——属于调试推理而非照着文档操作。  
- **为何直传**：主动查阅 Vercel **请求体限制**与社区案例，确认 multipart 不是无限容量。  
- **登录 Tab 配色**：在「品牌红」与「选中态可读性」之间尝试对比稿，而非固定一套设计 token。

### 4.4 学法小结（可复制）

1. **先复现、再缩小**：浏览器控制台 + Network + 服务端日志三方对齐。  
2. **官方文档优先于碎片博客**：Next/Prisma/Supabase 第一手说明更不易过时。  
3. **把决策写进仓库**（说明文档、技术说明书、过程记录）：迫使自己想清「理由」，也方便未来的自己答辩。

## 五、反思

- **真实使用暴露的问题**（滚动、保存感、上传无反馈）往往多于静态代码审查；小团队也应尽早做「走一遍完整任务流程」的冒烟。  
- **分布式与单机假设**必须在文档里写清（如 SSE 总线），否则后来者会误以为上线加机器即可线性扩展。  
- **AI 工具**：开发阶段用大模型辅助读文档、起草段落、拆解报错都很高效，但**架构取舍与安全边界**仍需人工拍板；产品内的 AI（OpenRouter）则是另一层「运行时依赖」，需在未配置时优雅降级。
