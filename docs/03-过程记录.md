# 03-过程记录

**目的**：记录真实落地中的判断、取舍与学习轨迹（正文主体以汉字为主，控制在约 500 字以内）。

---

## 落地感受（必选）

代码推到 GitHub 后用 Vercel 部署，体会到「提交 → 构建 → 访问」闭环很快，但**环境变量**（`DATABASE_URL` 池化与 `DIRECT_URL` 直连分离、`NEXT_PUBLIC_APP_URL` 与浏览器地址一致）一旦偏差，就会出现迁移超时或登录跳转异常；对照 `DEPLOY.md` 逐项核对后问题解决，也让我养成**先读部署文档再改线上**的习惯。真实使用中还收到界面类反馈（侧栏滚不到底、找不到保存按钮），说明**可用性不能只靠开发者自测**。

## 迭代调优（加分）

根据使用反馈陆续改了：任务详情侧栏用 flex + `min-h-0` 解决滚动；增加显式「保存」；私聊统一到消息中心便于查记录；「我的任务」补充状态筛选与日历视图。原则是**只改必要范围**，避免顺手大重构拖慢交付。

## 重要决策

- **技术栈**：Next.js（App Router）+ Prisma + PostgreSQL，兼顾接口与页面一体、类型与迁移可追溯；托管场景放弃 SQLite。
- **认证**：JWT + HttpOnly Cookie，实现简单、无服务端 Session 存储依赖。
- **实时**：SSE 推送私聊与项目同步，相对 WebSocket 更易在 Serverless 环境落地，代价是语义不如 WS 丰富，当前需求可接受。
- **AI**：OpenRouter 统一出口，便于换模型与密钥集中管理；未配置时功能降级而非阻断整站。
- **交互**：任务字段多为即时 PATCH，补充手动保存以降低「不知是否保存」的焦虑；私信进消息中心，与任务讨论区分职责。

## 学习过程

**会的**：沿用 React 组件拆分、REST 式 API、`fetch` 与表单交互。**新学的**：Prisma `migrate deploy` 与 Pooler/直连的关系、Next 服务端组件与 `"use client"` 边界、SSE 客户端订阅模式。**学法**：官方文档 + 构建/运行时日志定位错误；**主动探索**：组织维度私信列表、URL `?peer=` 深链打开会话、复制项目 ID 降低协作摩擦。
